# Voice & style

> How our product docs and stakeholder comms should read. Keep it short and concrete.

## Principles
- Lead with the decision or ask; put rationale after.
- Plain language over jargon. Define a term once, then use it consistently (see glossary).
- Active voice. Short sentences. No filler ("in order to", "it should be noted that").
- Quantify where possible: "cuts review time ~40%" beats "much faster".

## Document defaults
- PRDs follow `prds/_TEMPLATE.md` exactly.
- Stakeholder updates: TL;DR → RAG status → progress vs goals → risks → asks → next.
- Tone: direct, candid about risk, never overselling.

## Examples
- Prefer: "Ship the export button to Pro users by July 15. Risk: depends on the billing
  API freeze lifting."
- Avoid: "We are excited to potentially explore the possibility of an export capability
  in the near future."
